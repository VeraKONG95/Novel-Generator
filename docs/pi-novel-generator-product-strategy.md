# 基于 Pi 的小说生成器产品策略

## 1. 文档说明

- 产品形态：本地桌面应用
- 使用对象：个人小说作者
- 创作语言：首版专注中文
- 创作范围：支持所有小说类型
- 产品角色：Pi 是共同创作者，作者保留方向控制权和最终决定权
- 交付目标：经过基础自检的高质量初稿，不承诺无需审阅即可直接发表

本文档汇总产品定位、小说生成、人物生成、大纲规划、长篇记忆、独立评审，以及 Pi 创作环境的完整策略。

## 2. 产品定位

这不是无人值守的小说生产工具，也不是面向出版社团队的协作平台。

核心定位是：

> 作者负责创作方向、重要决定和最终审美判断；Pi 负责策划、生成、整理、检查和持续推进。

产品优先解决以下问题：

- 作者只有模糊想法，不知道如何形成完整故事。
- 长篇创作中人物、时间线、伏笔和世界规则逐渐失控。
- 大纲过于僵硬，或者完全没有规划。
- 模型只能完成单次生成，不能持续理解整个项目。
- 修改过程缺少记录，容易覆盖已经确认的内容。
- 已有小说缺少结构化整理，难以安全续写。

## 3. 产品原则

### 3.1 作者拥有最终决定权

Pi 可以提出方案、生成候选稿和说明影响，但不能擅自改变已经确认的故事方向。

### 3.2 正式正文是最终依据

人物状态、时间线、剧情进度和伏笔记录都属于辅助整理结果。如果它们与已确认正文冲突，以正文为准。

### 3.3 重要修改必须确认

覆盖已有大纲、人物核心设定或正式章节前，必须由作者确认。整理类信息可以自动更新，但必须透明、可编辑、可重新生成。

### 3.4 失败必须如实说明

模型请求失败、返回不完整或拒绝生成时，产品必须明确显示真实结果，不能用模板内容冒充成功。

### 3.5 项目文件开放可控

小说项目使用普通文件组成的创作空间。作者可以在应用之外编辑、搜索、备份和使用版本管理工具。

## 4. 核心创作流程

完整流程如下：

> 创作章程 → 核心人物 → 全书蓝图 → 当前阶段计划 → 近期章节计划 → 场景生成 → 完整章节 → 作者确认 → 更新故事记忆

作者需要在四个节点确认：

1. 故事核心：题材、主角、核心冲突。
2. 全书蓝图：结局方向、主要人物命运、阶段安排。
3. 当前阶段：阶段目标、关键转折、人物变化。
4. 章节成稿：确认后才成为正式正文。

同一章节内部，Pi 可以自行拆分场景、生成内容和完成基础整理，不需要频繁打断作者。

## 5. 创作章程策略

### 5.1 `AGENTS.md` 是项目规则入口

每个项目使用 `AGENTS.md` 作为唯一规则入口。它可以直接包含创作规则，也可以索引项目内的其他普通说明文件。

应用不提供技能安装、插件管理或自定义方法系统。所有扩展都通过普通文件组织，并由 `AGENTS.md` 建立索引。

### 5.2 新建项目时引导生成

新建小说时，Pi 通过一次简短访谈生成 `AGENTS.md` 初稿。访谈只询问真正影响创作的内容：

- 想写什么故事
- 面向什么读者
- 希望读者获得什么感受
- 叙事视角和语言倾向
- 计划篇幅和创作模式
- 明确禁止出现的内容或套路
- 必须遵守的世界规则
- 哪些决定必须由作者确认

信息不足时允许先建立最小版本，后续继续补充，不强迫作者一次回答大量问题。

### 5.3 规则冲突处理

当临时要求与创作章程或已确认剧情冲突时，Pi 必须：

1. 明确指出冲突位置。
2. 说明可选处理方向。
3. 说明每种选择影响哪些人物、伏笔和后续章节。
4. 等待作者确认。
5. 确认后统一修改相关内容。

Pi 不能悄悄忽略规则，也不能擅自修改历史。

## 6. 小说生成策略

### 6.1 三种创作模式

#### 规划型

提前确定完整故事结构和人物走向，适合推理、悬疑、历史以及结构复杂的作品。

#### 平衡型

确定全书方向，详细规划当前阶段和近期章节。它是默认模式。

#### 探索型

只锁定人物核心、当前冲突和近期方向，让故事在写作中自然生长，适合重人物、重氛围的作品。

三种模式只改变规划深度，不降低人物一致性、事实连贯和作者确认要求。

### 6.2 滚动大纲

产品不一次性把几百章全部写死，而是采用“远处定方向，近处做详细”的方式：

- 全书层面：确定主题、核心矛盾、主要人物命运和结局方向。
- 当前阶段：详细规划目标、关键转折和人物变化。
- 近期章节：详细规划接下来三到五章。
- 当前章节：拆解到具体场景。

每章确认后，Pi 根据实际成稿调整后续计划，但不能擅自改变已经确认的全书方向。

### 6.3 章节生成

Pi 内部按场景生成，最终按完整章节交付：

1. 明确本章剧情目标和情绪变化。
2. 拆分场景。
3. 确定每个场景的视角、地点、人物、冲突和结果。
4. 按顺序生成，并把前一场景结尾带入下一场景。
5. 合并后处理重复、跳跃、人物声音和整体节奏。
6. 形成完整章节，交给作者确认。

作者默认只面对完整章节。需要精细修改时，才展开到具体场景。

### 6.4 文风学习

作者可以提供自己的样章或参考文本。Pi 将参考内容整理成可确认的文风说明，而不是持续照搬原文表达。

文风说明包括：

- 句子长短和段落密度
- 叙述距离与视角
- 对话比例
- 描写重点
- 情绪强度
- 用词习惯
- 节奏和留白方式
- 明确禁止的表达

作者确认后，这些规则写入创作章程。

## 7. 人物生成策略

### 7.1 核心人物先确定

开书前优先确认：

- 主角
- 核心对手
- 最重要的关系人物
- 推动主线的关键人物

其他人物根据剧情需要逐步加入，不要求在开书前一次性创建全部角色。

### 7.2 人物必须具备完整驱动力

每个核心人物至少包含：

- 身份和关键经历
- 核心欲望
- 恐惧和创伤
- 秘密
- 能力与限制
- 行为底线
- 人物声音
- 主要关系
- 阶段变化
- 最终可能走向

不能只使用几个性格标签代替人物塑造。

### 7.3 新人物加入前检查必要性

生成配角或阶段性人物前，Pi 必须说明：

- 这个人物为什么必须存在
- 承担什么剧情作用
- 是否与现有人物功能重复
- 何时退出、改变或回归

作者确认后，人物才进入正式人物库。

### 7.4 稳定核心与动态状态分离

Pi 不能自动改变：

- 身份
- 关键历史
- 核心欲望
- 行为底线
- 长期秘密
- 作者锁定的事实

Pi 可以自动更新：

- 当前目标
- 情绪与身体状态
- 所在地点
- 已掌握的信息
- 人物关系
- 阶段性判断

重大人物转变必须说明触发事件、前文铺垫、关系影响和后续影响，并由作者确认。

## 8. 已有小说导入策略

产品同时支持“从零创作”和“导入续写”。

导入已有作品后，Pi 不能立即续写，必须先完成读稿建档：

- 识别主要人物及关系
- 整理世界规则
- 建立时间线
- 提取关键事件
- 找出未解决冲突
- 整理未回收伏笔
- 总结现有文风
- 推断创作章程

作者确认或修正整理结果后，Pi 才能继续创作。

首版支持纯文本和 Markdown。

## 9. 长篇记忆策略

### 9.1 章节确认后自动整理

每章正式确认后，Pi 自动记录：

- 新出现或被确认的事实
- 人物目标、情绪、位置、伤势和已知信息
- 人物关系变化
- 时间线推进
- 新埋下、推进或回收的伏笔
- 主线和支线进度
- 尚未解决的冲突

更新结果以简短变更记录展示，作者可以修正。

### 9.2 正文优先

如果整理结果与正式正文冲突，以正式正文为准。所有记忆都必须可以重新整理，不能成为无法检查的隐藏状态。

### 9.3 按需读取

Pi 每次任务只读取真正相关的内容：

- 创作章程
- 当前大纲和章节目标
- 本章相关人物状态
- 当前活跃的剧情和伏笔
- 最近章节摘要
- 上一章结尾
- 与当前任务直接相关的历史事件

不能每次把整本小说全部交给模型。

### 9.4 规模目标

首版按单本约 300 章、150 万字稳定运行设计，不设置硬性上限。

- 使用约 30 章检查连续创作质量。
- 使用 300 章项目检查读取、搜索和运行稳定性。

## 10. 独立评审策略

独立评审类似代码评审，由作者主动执行，不自动触发。

评审范围包括：

- 当前场景
- 当前章节
- 当前阶段
- 整本作品

评审只报告问题，不直接修改正文。每个问题包括：

- 问题位置
- 严重程度
- 违反了哪条创作章程或既有事实
- 为什么会影响阅读
- 建议的修改方向
- 可能受影响的后续内容

作者可以选择忽略、稍后处理、要求 Pi 提供修改稿，或者自行修改后重新评审。

## 11. Pi 创作环境

### 11.1 单 Pi 方案

整个产品只使用一个 Pi。策划、人物、大纲、写作、改写和评审通过不同任务切换，不建设多个智能体。

这样可以减少信息传递错误、重复消耗和事实不一致。

### 11.2 文件夹式创作空间

每个小说项目是一个独立文件夹，主要包含：

- `AGENTS.md`
- 全书蓝图
- 当前阶段大纲
- 章节计划
- 人物档案
- 每章正文
- 故事事实
- 时间线
- 伏笔记录
- 作者提供的参考资料

创作章程、大纲、人物档案和章节正文主要使用 Markdown。故事状态、索引和任务记录由应用维护。

### 11.3 外部编辑与版本管理

作者可以在应用外直接修改创作空间文件。

- 如果应用内没有未保存修改，自动刷新。
- 如果应用内外同时修改同一内容，不能自动覆盖，必须让作者选择。
- Pi 每次开始任务前重新读取相关文件。
- Pi 写入前再次确认文件没有被外部改变。
- 不完整或格式错误的文件先提示修复，不直接破坏项目。

应用不提供剧情分支和版本管理界面。作者可以通过 Git 或类似工具管理分支、比较修改和恢复历史版本。任何时候只维护一条正式故事线。

### 11.4 Pi 的权限边界

Pi 可以：

- 读取当前创作空间
- 搜索人物、事实、伏笔和章节
- 创建候选大纲、人物和章节
- 提交修改建议
- 在作者确认后写入正式内容
- 自动更新可以重新整理的故事记忆

Pi 不可以：

- 访问其他项目
- 访问创作空间外的文件
- 运行任意系统命令
- 自行执行版本管理命令
- 删除或覆盖未确认的正式正文
- 主动联网搜索资料

### 11.5 模型配置

所有项目共用一个全局配置文件，内容包括：

- 兼容 OpenAI 协议的服务地址
- 模型名
- 访问密钥

产品不提供额度、预算或官方模型服务。

首次使用时应检查模型能否配合 Pi 完成连续任务。只能普通聊天、不能调用应用能力的模型，不能启用完整创作流程。

访问密钥采用明文保存，因此全局配置文件必须放在小说项目之外，并明确提醒作者不要将它加入版本管理。

### 11.6 指令加载顺序

每次任务开始时，Pi 按以下顺序理解任务：

1. 产品不可突破的安全和权限边界。
2. 当前项目的 `AGENTS.md`。
3. `AGENTS.md` 索引的相关说明文件。
4. 当前任务对应的固定要求。
5. 当前任务需要的人物、剧情、章节和记忆。
6. 作者本次指令。

如果这些内容互相冲突，Pi 不能自行决定，必须说明影响并等待作者确认。

### 11.7 固定任务

产品同时支持自然语言和固定任务。固定任务至少包括：

- 创建项目
- 导入小说
- 生成人物
- 生成全书蓝图
- 规划当前阶段
- 规划下一章
- 写下一章
- 改写指定内容
- 独立评审
- 重新整理故事记忆

自然语言请求最终也要归入这些受控任务，避免 Pi 随意发明操作。

### 11.8 任务过程展示

任务运行时展开显示：

- Pi 对任务的理解
- 执行计划
- 正在读取的项目材料
- 正在执行的动作
- 面向作者的判断依据和取舍说明
- 发现的冲突
- 准备修改的内容
- 最终结果

完成后，过程自动收缩为“已处理 + 用时”，只保持最终内容展开。作者点击处理条后，可以查看完整过程。

### 11.9 运行控制

任务执行期间不能追加要求或临时改变方向。作者需要等待任务结束后再提出下一轮修改。

任务执行期间允许停止：

- 当前任务终止。
- 未完成内容不写入正式作品。
- 已完成的候选内容可以保留在处理记录中。
- 状态显示为“已停止”，不能显示为完成。
- 作者修改要求后重新执行。

### 11.10 异常恢复

应用在任务中途关闭后，重新打开时不自动继续。

应用需要展示：

- 上次执行到哪一步
- 已经生成了哪些内容
- 哪些内容尚未正式写入
- 任务为什么中断
- 重新执行和放弃任务两个选择

这样可以避免重复生成和误写正式内容。

### 11.11 对话组织

对话按任务拆分，例如：

- 全书蓝图讨论
- 人物设计
- 当前阶段规划
- 某一章生成
- 某次改写
- 某次独立评审

所有对话共享同一个正式项目，但对话记录只用于理解讨论过程，不能成为正式故事事实的唯一来源。

## 12. 产品界面策略

当前三栏结构可以继续保留，并重新明确职责：

### 左侧：任务与对话

- 展示任务历史
- 按任务组织对话
- 区分生成、改写和评审
- 显示完成、停止、失败和等待确认状态

### 中间：Pi 工作区

- 展示任务理解、计划和执行过程
- 展示候选稿和评审结果
- 提供确认、拒绝、停止和重新执行
- 任务完成后收缩处理过程，保留最终结果

### 右侧：正式作品

- 创作章程
- 人物档案
- 全书和阶段大纲
- 章节目录与正文
- 故事状态和伏笔

只有经过作者确认的内容才能进入正式作品区。

## 13. 当前应用的演进方向

现有生成、续写、改写、记忆、保存和导出能力继续复用，由 Pi 统一安排。

主要调整包括：

- 用真实 Pi 替换普通对话中的示意回复。
- 将分散按钮收敛为统一任务。
- 将单一项目文件迁移为文件夹式创作空间。
- 让人物状态、剧情进度和伏笔随确认章节持续更新。
- 将修改稿继续保留为候选内容，确认后再写入正式作品。
- 明确区分模型成功、模型失败和内容不完整。
- 保留旧项目自动迁移能力。
- 保留当前保存、自动保存和导出流程。

## 14. 首版不做的内容

- 多智能体
- 应用内剧情分支
- 应用内版本管理
- 主动联网研究
- 云同步
- 多人协作
- 多语言创作
- 模型额度管理
- 插件和技能安装
- 自动独立评审
- 任务运行中追加要求
- 意外退出后的自动续跑
- 产品自有商业收费体系

## 15. 内容边界

产品不增加额外的题材审查体系。

内容边界来自：

- 法律和基本安全要求
- 作者所配置模型服务的规则
- 作者在 `AGENTS.md` 中定义的禁区和尺度

如果模型拒绝生成，应用必须如实显示原因，不能偷偷换写法，也不能用模板冒充成功。

## 16. 实施路线

### 第一阶段：创作空间

- 建立文件夹式项目
- 使用 Markdown 保存正文和创作资料
- 支持 `AGENTS.md`
- 支持外部修改感知
- 处理内外修改冲突
- 迁移旧项目

### 第二阶段：Pi 主流程

- 接入单 Pi
- 建立固定任务
- 限制访问权限
- 按任务保存对话
- 展示计划、动作和判断依据
- 支持过程折叠
- 支持停止和异常恢复

### 第三阶段：生成体系

- 新书访谈与创作章程
- 核心人物生成
- 三种创作模式
- 滚动大纲
- 场景式章节生成
- 文风学习
- 已有小说导入与建档

### 第四阶段：长篇能力

- 人物状态更新
- 时间线维护
- 主线和支线进度
- 伏笔记录
- 记忆变更记录
- 手动独立评审
- 按需读取长篇资料

### 第五阶段：实际验证

- 使用多种中文题材完成完整创作流程
- 使用约 30 章验证连续创作质量
- 使用 300 章项目验证读取、搜索和运行稳定性
- 验证外部修改冲突
- 验证停止和异常退出
- 验证旧项目迁移
- 验证保存、自动保存和导出
- 确认不存在正文丢失和静默覆盖

## 17. 首版验收标准

首版完成需要同时满足：

- 新建小说可以完成章程、人物、蓝图、阶段计划、章节计划、正文生成、独立评审、修改和确认。
- 导入小说可以完成读稿建档、作者确认和继续创作。
- 每个项目都是可外部编辑和版本管理的文件夹。
- 外部修改能被发现，同一内容冲突时不会自动覆盖。
- Pi 只能操作当前创作空间，不能运行任意命令。
- 任务过程实时展开，完成后收缩为“已处理 + 用时”。
- 执行中不能追加要求，但可以停止。
- 应用意外关闭后不会自动继续，也不会误写正式内容。
- 正式章节确认后，人物状态、剧情进度和伏笔自动整理。
- 模型失败不会被伪装成成功。
- 旧项目可以迁移，正文、保存和导出不能丢失。
- 约 30 章连续创作测试通过。
- 300 章规模运行测试通过。
